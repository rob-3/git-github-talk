What is Git?

Git is a version control system created by Linus Torvalds that is commonly used for software.

  • Explain the concept of version control.
  • Better than zips and email.
  • Linus wrote the prototype in about 2 weeks because he didn't like any of the existing solutions.

Why should you care?

Git is by far the most popular version control system. It makes collaborating on software projects much easier and it is widely used across many companies and organizations who manage software.

Installing Git

  • https://git-scm.com/downloads
  • Follow instructions for your OS.
  • Give people a minute to get git installed.
  • Tell people they can create a GitHub account if they finish early.

Using Git

Creating a git repo

  • Create a new folder (or use an existing folder if we are adding git to an existing codebase)
  • Run git init
    • initializes the git repository
    • Git creates a .git folder to do all the magic

						$ mkdir hello_world
						$ cd hello_world
						$ git init
						Initialized empty Git repository in /home/rob/hello_world/.git/
						$ git status
						...
To start off, we need to create a directory (or folder) for our git repository and run git init. When we run git status, it helpfully tells us to create a file. Modern git has really nice messages in git status.

Adding some content

Git will immediately notice any change (including creation, deletion, or modification) to any file in the repository. 

						$ vim knighthacks.js
						$ git status
My shell prompt actually has nice built in git utilities, which show the current branch and whether I've changed anything.

The Index

  • Changes start off in the working tree.
  • The index is a staging area for changes we want to commit.
  • Use git add <file> to stage your changes.
  • A popular choice is git add . which adds every file in the current directory (folder).

						$ git add knighthacks.js
						$ git status
If you make a mistake, you can use git reset to unstage all of your changes.

Creating your first commit

A set of changes forms what git calls a commit. 

						$ git commit
Each time you make changes that you want to put in the project history, you will have to run git add and git commit.
  • If this is the first time you've used git, you'll probably get an error about setting your name and email. Git includes your name and email in every commit you make so that in a very large project it is possible to hunt down the specific person who wrote a piece of code. Go ahead and run the commands it tells you in the help.
  • You don't need to make a commit for a temporary change or test that you might not want to commit to (haha).

Writing a Commit Message

  • A commit message explains what the commit did, and potentially why.
  • Think of it like a comment, but not in the code.
  • Two parts:
    • Short summary (strict 80 char limit)
    • (Optional) extended message
  • Short summary should use the imperative mood.
    • "This commit will..."
  • Capitalize the first letter.
  • I see many people doing this improperly.
  • Short summary should contain the "what".
  • Extended message can contain the "why".
  • There should be a blank line between the summary and the extended message.
  • Main exception to imperative mood is "Initial commit"

The Log

git log is a useful command that displays the previous commits.
  • q quits
  • Arrow keys or vim bindings to scroll

Branching

  • We often want to maintain several separate versions of a codebase.
  • Feature branches
  • Personal branches
  • Stable release branches
  • ...
  • We might want to have a stable branch where we don't include untested changes, or maybe we are working on a new feature but it isn't finished yet.

Creating a new branch

The default branch is called master in git, but you can call new branches whatever you want. 

						$ git branch my_branch
						$ git checkout my_branch
						$ git status
  • git branch creates a branch but doesn't switch to it.
  • You can run git branch to see all branches.
  • Branches are implemented with pointers, so git branch is very efficient and doesn't copy anything.
  • git status and git log at this point will help you see what has happened.

Combining Branches

There are two main ways to combine branches:
  • Merge
  • Rebase
I will cover merge first and then rebase at the end if there is time. Rebases can be highly complicated and have the potential to do much more damage to your codebase.

Fast-Forward Merge

  • If only one branch has added commits since the split, git will do a fast-forward merge.
  • This simply points the current branch pointer to the latest commit on the secondary branch.
  • There will be no merge commit.

						$ git checkout master
						$ git merge my_branch

Normal Merge

  • If the branches have diverged, git will attempt to reconcile their differences.
  • Git always creates a merge commmit to signify the merge.
  • You might get a conflict if the two branches edited the same files or lines of code.
    • Git is pretty good at doing simple merges, but if there is a conflict, it won't know what to do. How you resolve a conflict depends greatly on the specific situation, and is something you can only learn with time.

    What is GitHub?

    • GitHub is a hosting provider for git repositories.
    • It is the most popular option, but there exist competing services like GitLab and Bitbucket that work the same way.
    • You can even self-host git on a server you own.

    Creating a GitHub account

    Head over to github.com and click sign up (top right corner).

    Git Architecture

    GitHub is just a server where git repositories can live and be accessed and written to by multiple people. Although git is decentralized and can be used in a peer-to-peer fashion, in practice, nearly everyone follows a simple client-server architecture.

    Linking your repo to GitHub

    • Create a new repository on GitHub
    • Navigate to your repository home page and press the green button to get the https link to your repository. 
    • 
								$ git checkout master
								$ git remote set-url origin <link>
								$ git push
    git push will give an error if the current branch doesn't exist on the server. Just follow the instructions and you'll be good. You may also be prompted to sign in to GitHub.

    Synchronizing with the server

    • Git stores a local copy of the state of the remote server in origin.
    • git fetch --all will synchronize origin with the server.
    • Then, you can run git merge origin/master to get the changes from the server into your local branch.

    git push and git pull

    • git status will now tell you if you your branch is ahead or behind origin.
    • If you're ahead, you can use git push to update the server.
    • If you're behind, you can use git pull to update your local repo.
    This is where that configuration option comes in. By default, git pull will do a merge, but I prefer it only to merge if it is a fast-forward. Otherwise, I would like to consider my options before anything is done automatically. git pull --rebase can be used to automatically do a rebase.

    GitHub Features Demo

    git reset

    • git reset <commit> will undo all of the commits since the given commit and put the changes in your working tree (red text).
    • git reset --hard <commit> will undo commits since the given commit and drop the changes entirely.
    • Both of these commands are commonly used with no arguments to indicate the current commit (HEAD)
    • Many people believe you can't recover from a bad git reset --hard, but you actually can with git reflog.

    .gitignore

    .gitignore is a file you can create so that git will ignore a file or folder from its tracking. It is useful for things like node_modules or log files.
